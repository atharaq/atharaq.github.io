Find the mistakes

Problem: given a String name, return the number of times the character c appears in it. What are all the mistakes in the following?

public static int numAppearances(String name, char c) {
  String name = "bob";
  char c = "b";
  int n = 0;
  for (int i = 0; i < name.length(); i++) {
    if (name.charAt(i) == c) {
      System.out.println(c);
      n++;
    }
  }
  System.out.println(n);
}

Immutable GaussianInteger

To re-design GaussianInteger to be an immutable type, first we can try to ensure that the instance variables can’t be changed: e.g. make them all final! First try that.

Any errors? They tell us what we need to change:

  • add and multiply.
  • Instead of making them void, have them return new GaussianInteger objects.
  • Now we don’t change our original object!

More tips

findByName:

  • For \(i = 0\) to the length of the list:
    • If the name of student \(i\) equals the name that you’re looking for, print it out
    • Can assume there is only one Student with that title, so you can return at this point.
  • If you don’t find a Student by that title, output “There are no students named … in the course.”
    • How can you tell if there is no student with that name?

findAllAboveGrade

  • For \(i = 0\) to the length of the list:
    • If student \(i\) has a grade that is above the grade you’re looking for, print it out
    • In this case, there may be more than one student above that grade.
  • If you don’t find any students with that grade, output “There are no students with a grade above … in this course.”
    • How can you tell if there is no student with a high enough grade?
    • A little harder than the previous one!

Kinds of errors

  • Compiler errors: syntax errors, type mismatches, etc. These are caught by the compiler.
  • Run-time errors: usually Exceptions, but also things like OutOfMemoryError.
  • “Bugs” or logic errors: when the program doesn’t do what you wanted it to do. Focus today: these types.

Examples for today are taken from Bloch and Gafter, Java Puzzlers: Traps, Pitfalls and Corner Cases (2005).

Debugger

Modern IDEs have built-in tools to help us debug.

  • Click on the “side” of a line to add a breakpoint.
  • Use the debugger (replit: left side menu; on IntelliJ: Run menu -> “Debug…”)
  • The program runs until it hits the breakpoint, and then lets us continue execution one line at a time.
  • The debugger shows us a memory diagram, the values of all the variables at a particular moment.
  • What is the variable in the last println method? Why?
  • Also: notice anything in IntelliJ?

Lessons

There are two main lessons here:

  1. Precedence rules are tricky. Use parentheses when worried about precedence rules. It’s not clear that the ‘+’ symbol concatenates two strings first, and then the == symbol checks for equality after, but that’s what happened!
  2. Reference equality (==) is usually not what you want to use for non-primitive types. Use the .equals method!

InTheLoop

Look at the InTheLoop program:

public static final int END = Integer.MAX_VALUE;
public static final int START = END - 5;

public static void main(String[] args) {
    int count = 0;
    for (int i = START; i <= END; i++)
        count++;
    System.out.println(count);
}
